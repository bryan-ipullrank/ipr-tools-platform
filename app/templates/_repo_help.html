Required plugin repo structure

A plugin's source is a Git repository, not a file the portal serves. When a developer runs /plugin install <name>@{{ marketplace_name }}, Claude Code clones the whole repo and looks for a manifest at .claude-plugin/plugin.json. It is not a bare .skill file — skills live unpacked under skills/.

Minimum viable repo

{% raw %} 
your-plugin-repo/
├── .claude-plugin/
│   └── plugin.json        # REQUIRED — only "name" is mandatory
└── skills/
    └── my-skill/
        └── SKILL.md       # the skill itself (unpacked, not a .skill file)
{% endraw %}

plugin.json (manifest)

{% raw %} 
{
  "name": "my-plugin",          // kebab-case; the component namespace
  "version": "1.0.0",           // optional but recommended (else uses commit SHA)
  "description": "What it does"
}
{% endraw %}

Optional components (all at the repo root)

{% raw %} 
skills/<name>/SKILL.md   # model-invoked skills
commands/*.md            # slash commands
agents/*.md              # subagents
hooks/hooks.json         # event hooks
.mcp.json                # MCP servers
{% endraw %}

Only plugin.json goes inside .claude-plugin/; everything else sits at the repo root.

Two names, two jobs

The plugin name in this catalog is the install id (name@{{ marketplace_name }}). The name in the repo's plugin.json is the component namespace (e.g. /my-plugin:my-skill). They don't have to match.

Access

The repo is private, so an installing developer needs GitHub access to it. If a clone fails, use Request access on the plugin to ping the owner.

Full reference: code.claude.com/docs/en/plugins-reference
